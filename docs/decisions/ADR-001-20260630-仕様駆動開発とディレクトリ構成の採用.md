# ADR-001: 仕様駆動開発とディレクトリ構成の採用

- **Date**: 2026-06-30
- **Status**: Accepted

## コンテキスト

Markdownビューワーの開発を開始するにあたり、開発手法とディレクトリ構成を決定する必要があった。
開発にはClaude CodeとGemini（Antigravity）を併用する予定であり、両AIに対して一貫したコンテキストを提供する仕組みが求められた。

## 決定事項

- 開発手法として**仕様駆動開発**を採用する。
- ディレクトリ構成はAWSのKiroのアプローチを参考に、`.kiro/` を中心とした構成とする。
- AI共通の開発規約・コンテキストは `.kiro/steering/` に集約し、`CLAUDE.md` と `GEMINI.md` はここを参照するだけの薄いラッパーとする。
- 意思決定は `docs/decisions/` にADR形式で記録する。
- 改行コードはLFに統一し、`.gitattributes` で強制する。

## 理由

- **仕様駆動開発**: 要求 → 設計 → タスクの順序で進めることで、実装前に合意形成できる。
- **`.kiro/steering/` への集約**: ルールの二重管理を防ぎ、どのAIツールが読んでも同一の挙動になる。
- **ADR**: 「なぜそう決めたか」を後から追跡できる。設計の背景が失われることを防ぐ。
- **Kiro参考**: KiroはAWSが提案する仕様駆動開発ツールであり、AIとの協調開発のベストプラクティスが反映されている。

## 影響・トレードオフ

- `.kiro/` はKiro専用ツールを使わなくても機能する（ただのディレクトリ）。Kiro導入時の移行コストは低い。
- Geminiの `GEMINI.md` はClaude Codeの `@import` 記法に相当する機能がないため、参照先を明示するテキスト形式とした。
- コーディング規約はアーキテクチャ確定後に `steering/conventions.md` へ追記する。
