# ADR-009: ファイルアクセスの信頼境界をRust側（AllowedRoots）に置く

- **Date**: 2026-07-08
- **Status**: Accepted

## コンテキスト

本アプリはローカルのMarkdownファイルを閲覧するTauri v2アプリで、ファイルアクセスはRust側のコマンド（`read_file` / `read_directory` / `read_image_data_url`）が担う。`html: true` のmarkdown-itで生HTMLを描画し、Mermaid・KaTeX・Shikiなどリッチなレンダリングを行うため、**悪意あるMarkdownファイルが攻撃者の入力になりうる**（信頼できない `.md` を開くケースを想定する必要がある）。

当初の実装（セキュリティ診断以前）は、各コマンドがフロントエンドから受け取った `path` を無検証で読み込んでいた。診断を受けた最初の修正では、各コマンドに `root` 引数を追加し、`path` が `root` 配下にあることをcanonicalize（シンボリックリンク解決込み）して検証する方式にした。

しかしこの方式には根本的な弱点が残っていた。**境界チェックの信頼起点がWebView（フロントエンド）側にある**点である。`root` を呼び出し側が自由に指定できるため、万一WebView内でスクリプトが実行された場合（生HTML描画・Mermaid出力・依存ライブラリのXSS等が起点になりうる）、攻撃者は「読みたいファイルの親フォルダを `root` に指定」して `invoke("read_file", { path: "C:/Users/.../secret", root: "C:/Users/..." })` のように呼べば、canonicalizeチェックを常に通過できてしまう。これでは任意ファイル読み取り → CSP次第で外部送信、という経路が残る。

## 決定事項

**ファイルアクセスの信頼境界を、フロントエンドではなくRust側のmanaged state `AllowedRoots` に一元化する。** コマンドから `root` 引数を廃止し、信頼済みルートの集合をRustが保持・判定する唯一の情報源とする。

実装は `app/src-tauri/src/commands/file.rs`。

### 主要な設計

1. **`AllowedRoots`**: ユーザーが明示的に開いたフォルダールートと単一ファイル（いずれもcanonicalize済み）の集合。通常のMarkdown・フォルダーは従来どおりルート配下を許可し、ZIPはアーカイブファイルとの完全一致だけを許可する。`lib.rs` の `.manage()` で注入する。

2. **信頼登録はRustが確認できる操作に限定する**: WebViewから任意パスを無確認で登録できる`register_root(path)`は公開しない。ネイティブファイル/フォルダ選択、Rustが受信したドラッグ＆ドロップ、プロセスのCLI引数はRust側で直接登録する。保存済みパスやルート外リンク等は`authorize_path(path)`、ZIPは`authorize_archive_path(path)`がネイティブ確認を表示し、承認された場合だけ登録する。

3. **読み取りコマンドは `root` 引数を持たない**。`read_file` / `read_directory` / `read_image_data_url` は `path` をcanonicalizeし、`AllowedRoots` のフォルダールート配下（`starts_with`）、または認可済み単一ファイルと完全一致する場合のみ処理する。判定ロジックは `AllowedRoots::resolve` に集約する。

4. **`is_within_root` を `is_path_allowed(path)` にリネーム**。引数は `path` のみ。リンククリック時の事前判定に使う。

5. **WebViewの操作状態を認可根拠にしない**。フロントエンドのクリックハンドラー、ダイアログAPIの戻り値、復元トーストの状態は、WebView内でスクリプトが実行された場合に偽装できる。認可はRustが直接受け取ったOS入力、またはRustが表示したネイティブ確認の結果だけを根拠にする。`openMarkdownFile`自身は登録しない。

### 危険フォルダの拒否

信頼登録処理は、広すぎる範囲を信頼させないため以下を拒否する（Windowsを主対象とし、canonicalize後のverbatimパスでも正しく判定できるよう `Normal` コンポーネントのみで評価する）。

- ドライブ直下・ファイルシステムルート直下（例: `C:\`, `/`）
- Windowsシステムフォルダ（`Windows` / `Program Files` / `Program Files (x86)` / `ProgramData`）
- ユーザープロファイル直下（例: `C:\Users\<name>` そのもの。配下の `Documents` 等は許可）

### 限定的な信頼の永続化（2026-07-11改訂）

最後にネイティブ選択またはネイティブ確認で明示承認されたExplorerルート1件だけを永続化する。起動時にcanonicalize・ディレクトリ種別・危険ルート拒否を再検証し、有効な場合だけ`AllowedRoots`へ登録する。最近使った別フォルダーと別ルートのタブは従来どおり`authorize_path`のネイティブ確認を通す。

- **理由**: 毎起動時に前回のExplorerルートを再確認する摩擦を減らしつつ、信頼リストが無制限に蓄積する事故を避けるため。
- **保存境界**: WebViewのtauri-plugin-store汎用権限を廃止し、settings/tabs/recentは固定識別子のRustコマンドだけで扱う。永続信頼用ストアはRust内部からのみ到達でき、WebView状態を認可根拠にしない。
- **前提**: 本アプリはWindowsを主対象とし、用途は「フォルダを開いて閲覧する」が中心。ユーザープロファイル直下に単独ファイルを置いて開くようなケースは稀と判断した。macOS/Linuxのホームディレクトリ直下の扱いは主対象外であり、必要になった時点で再検討する。

## 関連するセキュリティ修正（同時対応）

信頼境界の変更と併せて、同一ブランチで以下も対応した。いずれも「信頼できないMarkdownが入力になる」前提での多層防御。

- **Mermaidエラー表示のHTMLインジェクション対策**: レンダリング失敗時に元コードを `innerHTML` へ直接埋め込んでいた箇所を `textContent` 挿入に変更。Mermaid本体の `securityLevel` も `loose` から既定の `strict` に戻し、HTMLラベル・clickディレクティブの素通しを止めた（`mermaid-post.ts`）。markdown-it出力は従来どおりDOMPurify（`sanitize.ts`）を通す。
- **CSPの最小化**: `csp: null` を廃し、`default-src 'self'` を基本に必要最小限へ絞った（`tauri.conf.json`）。リモート `connect` は使わないため `connect-src 'self'`、画像は `img-src 'self' data: https:`（平文 `http:` は除外）。
  - **`script-src` に `'wasm-unsafe-eval'` を追加**: Shikiのシンタックスハイライトはonigurumaを**WebAssembly**で動かすため、リリースビルドでCSPが厳格適用されると `WebAssembly.instantiate` がブロックされる。`'unsafe-eval'`（JSの `eval` を全面許可＝危険）ではなく、WASMのコンパイル/インスタンス化のみを許可する `'wasm-unsafe-eval'` を用いることで、`eval` を禁止したまま必要な機能だけを通す。dev環境ではTauriがCSPを緩めるため顕在化せず、リリースビルドで判明した。

## 不変条件: FSに触れる全コマンドが `resolve` を通す（2026-07-11 追記）

本ADRの信頼境界は、**フロントエンドから受け取ったパスでファイルシステムに触れるTauriコマンドは、処理前に必ず `AllowedRoots::resolve`（または `register` 相当の明示登録経路）を通す**という不変条件に依存する。この不変条件が1コマンドでも破れると、そこがWebView経由の任意パスアクセスの抜け穴になる。

後日のレビューで、初期実装以降に追加された2コマンドがこの不変条件から漏れていたことが判明し、修正した。

- `search_in_directory`（`search.rs`）: `root` を無検証で走査しMarkdown本文を返していた。`state.resolve(&root)?` を追加。
- `watch_path`（`watcher.rs`）: 任意パスの監視登録が可能だった。`roots.resolve(&path)?` を追加。

現在、この不変条件の対象コマンドと検証方法は以下のとおり（新規にFSへ触れるコマンドを追加する際は、必ず本表に従って `resolve` を通すこと。回帰防止のため各コマンドにテストを持たせる）。

| コマンド | ファイル | 検証 |
| --- | --- | --- |
| `read_file` | `file.rs` | `resolve` |
| `read_custom_css` | `file.rs` | `resolve` ＋ 拡張子・サイズ制限 |
| `read_directory` | `file.rs` | canonicalize済み起点＋候補ごとの`resolve`＋10,000件上限 |
| `read_image_data_url` | `file.rs` | `resolve` |
| `open_in_editor` | `file.rs` | `resolve` |
| `search_in_directory` | `search.rs` | canonicalize済み起点＋候補ごとの`resolve`＋走査/結果/一致件数上限 |
| `watch_path` | `watcher.rs` | `resolve` |
| `watch_directory` | `watcher.rs` | `resolve` |
| `watch_custom_css` | `watcher.rs` | `resolve` ＋ 拡張子 |
| `resolve_wiki_links` | `wiki.rs` | canonicalize済み起点＋候補ごとの`resolve`＋走査10,000件・入力1,000件上限 |

例外的にパス検証を持たないコマンド（`stat_path`・`get_cli_args`）は、返す情報がパスの存在有無・種別やプロセス引数に限られ、ファイル内容を読まない。任意パスの存在プローブになりうる点は許容範囲としてリスク受容する（用途上ゲートしづらいため）。

### 型レベル保証を採らない理由

「検証済みパスを表す新型（例: `TrustedPath`）をコマンド引数に受け取り、コンパイル時に検証漏れを防ぐ」案も検討したが採用しない。Tauriのコマンド引数はIPCから**managed stateにアクセスできない文脈でデシリアライズ**されるため、newtype自身に `AllowedRoots` 参照の検証を持たせられない。検証を伴わないまま「検証済み」を名乗る型はむしろ誤った安心を生む。よって本表の遵守と各コマンドのテストで不変条件を担保する。

## 影響・トレードオフ

- **メリット**: 信頼起点がRust側に固定されるため、WebViewが任意の `root` を捏造しても登録済みルート外は読めない。フロントとバックエンドで境界判定ロジックが二重化していた状態も解消し、判定はRust側の単一実装に集約された。
- **デメリット / 制約**:
  - ユーザープロファイル直下など危険フォルダに置いた通常のMarkdown単独ファイルは、現状そのままでは開けない。ZIPは親フォルダーを信頼しない単一ファイル認可のため、この制約とは独立して扱える。
  - 最後のExplorerルート1件は再起動後も信頼される。別の最近使ったフォルダーは再確認される。
- **IF変更**: `is_within_root` → `is_path_allowed` のリネームと各コマンドの `root` 引数廃止はフロント/バックエンド間のインターフェース変更。フロント側の呼び出し箇所はすべて追従済み。

## セキュリティ再レビューによる改訂（2026-07-11）

初版の`register_root(path)`はmanaged stateをRust側に置いた一方、WebViewから任意パスで呼び出せた。そのためWebView内でスクリプトが実行された場合、攻撃者が既知の機密パスを登録してから`read_file`等を呼べるという、初版が防ごうとした信頼起点の偽装が残っていた。

この問題を受け、以下へ改訂した。

- 公開`register_root`を廃止する。
- `pick_markdown_file` / `pick_folder` / `pick_custom_css`はRust側でネイティブダイアログを表示し、選択結果をRust内で登録してからWebViewへ返す。
- OSドラッグ＆ドロップは`on_window_event`、CLI入力は初回`setup`およびsingle-instanceコールバックでRustが直接受信したパスだけを登録する。WebViewから同名イベントを偽装しても登録は増えない。
- 最近使った項目、セッション復元、保存済みカスタムCSS、信頼範囲外リンクは`authorize_path`を使う。未認可の場合はcanonicalize後の対象フォルダをRustネイティブダイアログに表示し、ユーザーが承認した場合だけ登録する。
- CDPを使う開発時E2Eの`authorize_dev_path`はdebugビルドだけ登録を許可し、releaseビルドでは常に拒否する。
- `read_file`はMarkdown拡張子に限定し、10MiBを上限とする。メタデータ確認後も`take(MAX + 1)`で読込量を制限し、差し替えによる上限超過を防ぐ。
- DnD/CLIの1入力で扱うパスは32件までとし、フロントとRustの両方で上限を適用する。
- CLI引数はRust側で実行ファイル自身を除外してから最大32件に制限し、同じ配列を信頼登録とWebViewイベントへ使う。
- ネイティブダイアログは共有`NativeDialogState`で同時に1件だけ許可し、WebViewからの並列invokeによるダイアログ洪水を拒否する。
- ファイル内容を読むコマンドは、canonicalizeとopenの間の差し替えを考慮する。最初にopenしたハンドル自身の最終解決パスをOS API（Windows: `GetFinalPathNameByHandleW`、Linux: `/proc/self/fd`、macOS: `F_GETPATH`）で取得し、そのパスが`AllowedRoots`内の場合だけ同じハンドルから読む。この共通処理をMarkdown、カスタムCSS、画像、全文検索へ適用する。
- ディレクトリ列挙はcanonicalize済みの走査起点を使い、各候補も`resolve`で再検証する。ツリー・全文検索・Wiki候補の走査は最大10,000エントリ、全文検索は最大1,000ファイル・1ファイル最大1,000一致、Wiki解決入力は最大1,000件に制限する。
- 外部エディターの実行ファイルはRustネイティブpickerまたはRustネイティブ確認で認可し、`ExternalEditorState`へ正規化済みの完全パスを保持する。`open_in_editor`はWebViewからコマンドを受け取らず、認可済みstateだけを使い、対象ファイルもMarkdownに限定する。

## ZIP単一ファイル認可への改訂（2026-07-17）

ZIPを開く際に親フォルダー全体を信頼すると、確認画面の対象と実際の権限範囲が広くなりすぎる。ZIP内のMarkdown・画像はアーカイブ内部から読み取り、親フォルダーの別ファイルを参照する必要がないため、ZIPはcanonicalize済みアーカイブファイルとの完全一致だけを`AllowedRoots`へ登録する。

- ネイティブpicker、DnD、CLI入力はZIPだけを単一ファイルとして登録する。
- 最近使った項目・履歴・セッション復元は`authorize_archive_path`を使う。
- ネイティブ確認には「アーカイブへのアクセス確認」とアーカイブのフルパスを表示する。
- `resolve`と開いたファイルハンドルの最終パス検証は、フォルダールート配下または単一ファイル完全一致のどちらでも同じ強度で適用する。
