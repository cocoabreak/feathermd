# ADR-008: Shikiのシンタックスハイライト対応言語を限定する

- **Date**: 2026-07-08
- **Status**: Accepted

## コンテキスト

コードブロックのシンタックスハイライトにはShikiを使用している。Shikiは235言語（エイリアス込み346）のTextMate文法をバンドル可能だが、全言語に対応するか、使う言語だけに限定するかは選択の余地がある。

当初は`import { createHighlighter } from "shiki"`（"full bundle"）を使い、`langs`オプションで25言語に絞っていた。しかしこの指定は実行時フィルタでしかなく、ビルド出力には未指定の言語（`emacs-lisp`/`julia`/`rst`/`ruby`等）まで静的に含まれていることが判明した。これを受け、`shiki/core` + 言語ごとの個別import（fine-grained bundle）に切り替え、実際に使う言語だけをビルドに含める方式に変更した（現在は38言語に対応）。

この変更にはトレードオフがある。fine-grained bundleは言語を追加するたびに`engine.ts`へimportを1行足す必要があり、全言語対応（またはfull bundle）ならその手間が要らない。「配布サイズが多少大きくなるだけなら、全言語対応にしてメンテナンスフリーにした方が良いのでは」という疑問が出たため、実測で比較した。

## 検証結果

本番ビルド（`vite preview`）+ 実ブラウザでの計測。

| 構成 | クライアントJS合計 | Shiki初期化時間 |
|---|---|---|
| fine-grained bundle・38言語 | 約6.9MB | 約183ms |
| full bundle・同じ38言語 | 約6.9MB相当ではなく実質full bundle最小構成と同等（後述） | 約185ms |
| full bundle・全235言語 | 約13.5MB | **約3,057ms** |

- **配布サイズ**: full bundleは`langs`オプションで何言語を要求しても、パッケージの内部チャンク構成上ほぼ一定（約13.5MB）になる。25言語だけを要求していた変更前のmainも、全235言語を要求した場合も、ビルド出力はほぼ同じ約13.5MBだった。したがって「full bundleのまま対応言語を絞る」ことにサイズ面の恩恵はほぼ無く、サイズを削減するにはfine-grained bundle化が必須だった。
- **初期化時間**: 言語数を38→235に増やすと初期化（グラマー解析・正規表現コンパイル）だけで約2.9秒増加した。これはfine-grained/full bundleのどちらでも、要求した言語数にほぼ比例する（バンドル方式には依存しない）。

### 体感への影響（実機・tauri devアプリでの計測）

初期化コストが実際にユーザー体感へ影響するかを、「復元操作（リロード）から画面にコンテンツが表示されるまで」の時間で計測した。

| 構成 | 復元→表示（3回平均） |
|---|---|
| fine-grained bundle・38言語 | 約800ms |
| full bundle・全235言語 | **約3,290ms** |

`renderMarkdown()`は言語に関わらず`getHighlighter()`の完了を待ってから描画するため、この差（約2.5秒）はセッション復元時にそのまま体感される。「最近開いたファイル」のワンクリックオープンのようにOSのファイルダイアログを介さない操作も同様に影響を受ける。「ファイルを開く」ダイアログ経由であればダイアログ操作中の時間（体感で最低1〜2秒程度）にある程度隠れる可能性はあるが、操作の速いユーザーでは隠れきらない。全言語対応は「ファイルを開く操作の裏で初期化すれば実害なし」とは言い切れず、軽量ビューワーという方針とは相性が悪いと判断した。

## 決定事項

**Shikiの対応言語は用途に応じて選定した一覧に限定し、全言語には対応しない。** 実装は`app/src/lib/markdown/engine.ts`の`HIGHLIGHT_LANG_LOADERS`（この一覧が唯一の情報源）。

対応言語を追加したい場合は、同ファイルに`@shikijs/langs/<言語ID>`のimportを1行追加する（要ビルド）。

### 現在の対応言語（38言語、2026-07-08時点）

typescript, javascript, tsx, jsx, svelte, rust, go, python, java, c, cpp, csharp,
bash, sh, powershell, json, yaml, toml, xml, html, css, scss, sql, markdown, diff,
mermaid, cobol, dockerfile, dotenv, git-commit, git-rebase, graphql, http, ini,
kotlin, groovy, latex, log

このリストは`engine.ts`の変更に追従して古くなりうるため、正確な最新一覧は必ずコードを参照すること。

## 影響・トレードオフ

- **メリット**: 配布サイズを約半分（約13.5MB→約6.9MB）に抑えられる。初期化時間も全言語対応時と比べて約2.9秒短く、セッション復元等で体感される待ち時間の差としても約2.5秒確認済み
- **デメリット**: 新しい言語に対応するにはコード変更（import追加）とビルドが必要。ユーザーからの「この言語にも対応してほしい」という要望には即応できず、次回リリースまで待つ必要がある

### 将来の代替案: 軽量版とフル版のリリースを分ける

対応言語のリクエストが増え、都度のメンテナンスコストが無視できなくなった場合に検討する代替案。現行の「軽量版のみ」（本ADRの決定）に加えて、全言語対応の「フル版」を別ビルド・別配布物として用意する。

- 軽量版: 本ADR通りfine-grained bundle・厳選した言語一覧（初期化が速い、配布サイズが小さい）
- フル版: full bundle・全235言語対応（初期化に約3秒かかることを許容するユーザー向け）

実現には、ビルド時フラグ（`import.meta.env`による分岐、DEVフックと同様のdead-code-elimination）で2種類のビルド成果物を出し分ける仕組みと、Tauriのリリース・配布パイプラインを2系統に分ける対応が必要になる。現時点ではニーズが顕在化していないため見送るが、ユーザーからの言語追加要望が増えてきたら着手を検討する。
