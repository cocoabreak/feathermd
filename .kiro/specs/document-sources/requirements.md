# 要求定義: 読み取り専用ドキュメントソース (document-sources)

## 背景・目的

FeatherMDのExplorer・タブ・リンク・画像・全文検索・Wikiリンクは、現在すべてOSの
ネイティブファイルシステム上のパスを前提としている。ZIPをフォルダーとして閲覧する
ために各機能へZIP固有分岐を追加すると、将来7zやtar系を追加するたびに同じ変更が必要になる。

本specでは、ビューワーに必要な読み取り操作だけを「ドキュメントソース」として抽象化する。
v0.1.0ではネイティブファイルシステムとZIPを実装し、7z・tarおよび各種圧縮方式は将来の
バックエンドとして設計上のみ考慮する。

## スコープ

- 読み取り専用DocumentSource API
- NativeSourceへの既存処理の移設（既存動作を維持）
- ZipSourceによるZIP内Markdown・画像・ディレクトリの読み取り
- Explorer、タブ、相対リンク、画像、全文検索、Wikiリンク、履歴、セッション復元、監視のソース対応
- ZIP Bomb、パストラバーサル、暗号化・分割ZIP等へのfail-closedな処理

対象外:

- アーカイブへの書き込み・再圧縮
- ZIP内ファイルの一時展開と外部エディター連携
- 暗号化ZIP、分割ZIP
- 入れ子アーカイブの再帰展開
- 7z、tar、tar.gz、tar.zst等の実装
- 実行時に第三者バックエンドを読み込む動的プラグイン機構
- OSへ仮想ドライブとしてマウントする汎用VFS

---

## US-001: ネイティブフォルダーの既存動作を維持する

As a ユーザー、DocumentSource導入後も通常フォルダーを従来どおり閲覧したい。

受け入れ条件:

- [x] フォルダー選択、遅延展開、Markdown表示、相対リンク・画像が従来どおり動作する
- [x] `.gitignore`、隠しファイル設定、全文検索、Wikiリンクが従来どおり動作する
- [x] 個別ファイル・展開中ディレクトリの監視と削除状態が従来どおり動作する
- [x] 外部エディター、履歴、セッション復元が従来どおり動作する
- [x] AllowedRootsと開いたファイルハンドルの最終パス検証を維持する

## US-002: ZIPをフォルダーとして開く

As a ユーザー、ZIPを展開せずにExplorerからMarkdownを閲覧したい。

受け入れ条件:

- [x] 「アーカイブを開く」から`.zip`を選択できる
- [x] ZIPをExplorerのルートとして表示し、内部フォルダーを遅延展開できる
- [x] `.md` / `.markdown`と、それらを含む仮想ディレクトリを表示する
- [x] ZIP内Markdownを通常タブで表示できる
- [x] ZIP内の相対Markdownリンクとアンカーを解決できる
- [x] ZIP内の対応画像を既存の画像表示・拡大機能で表示できる
- [x] ZIP内Markdownの全文検索とWikiリンク解決が動作する
- [x] ZIPを最近使った項目から再度開ける
- [x] ZIPソースとZIP内タブをセッション復元できる

## US-003: ソースごとの能力差を正しく扱う

As a ユーザー、バックエンドで不可能な操作が誤って提示されないようにしたい。

受け入れ条件:

- [x] NativeSourceはwatch・externalEditor・respectGitignoreを提供する
- [x] ZipSourceはコンテナ監視を提供し、externalEditor・respectGitignoreを提供しない
- [x] ZIP内ファイルでは外部エディター項目を表示しない、または無効理由を明示する
- [x] ZIP本体変更時はインデックスを破棄し、Explorerと開いているZIP内タブを再読込する
- [x] 更新後に消えたZIPエントリはタブをdeleted状態にする

## US-004: アーカイブ入力を安全かつ有限に処理する

As a ユーザー、信頼できないZIPを開いてもファイルシステムやアプリ全体へ影響させたくない。

受け入れ条件:

- [x] ZIP本体は親フォルダーへ権限を広げず、明示的に許可されたアーカイブファイル単体だけを開く
- [x] 未認可ZIPの確認画面はアーカイブのフルパスと、ファイル単体へのアクセスであることを表示する
- [x] 絶対パス、NUL、ルート外へ解決されるパス、シンボリックリンクを拒否する
- [x] 正規化後に重複するエントリを拒否する
- [x] 暗号化ZIP・分割ZIP・未対応圧縮方式を明示的なエラーにする
- [x] エントリ数、単一ファイル展開サイズ、総展開宣言サイズ、実読込量、圧縮率に上限を持つ
- [x] Markdown 10MiB、画像20MiBの既存上限をZIP内でも維持する
- [x] アーカイブ内容をディスクへ展開しない
- [x] 失敗したZIPが他のソースや既存タブを壊さない

## US-005: 将来形式を追加できる境界を持つ

As a 開発者、7zやtar系を既存UIの全面改修なしに追加したい。

受け入れ条件:

- [x] Explorer・タブ・Viewerはバックエンド固有の実パス形式へ依存しない
- [x] ソースはRust側の静的レジストリーへ登録し、フロントはsourceIdと相対仮想パスを使う
- [x] バックエンド固有差はcapabilitiesと内部実装へ閉じ込める
- [x] v0.1.0のバックエンドはNativeとZIPだけとし、動的ロードは行わない

## 非機能要求

- ソース内パスは`/`区切りの正規化済み相対パスとし、OSパスと混在させない
- Rust側の検証をセキュリティ境界とし、フロント側検証だけに依存しない
- アーカイブI/Oと検索はWebViewメインスレッドをブロックしない
- 上限値は定数化し、境界値・悪性入力テストを持つ
- NativeSource移行とZipSource追加を独立して検証可能にする

## 未決定事項（設計フェーズで決定）

- [x] 汎用VFSか読み取り専用DocumentSourceか → DocumentSourceを採用
- [x] 動的trait/pluginか静的enumか → v0.1.0は静的enum dispatchを採用
- [x] フロントへ実パスを渡すか → sourceId + 相対VirtualPathを採用
- [x] ZIPを一時展開するか → 展開せず必要エントリだけ読み取る
- [x] v0.1.0対応形式 → Native + ZIP
- [x] 7z/tar系 → 将来バックエンド。v0.1.0では実装しない
