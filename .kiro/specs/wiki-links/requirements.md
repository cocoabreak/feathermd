# 要求定義: Wikiリンク (wiki-links)

## 背景・動機

Obsidianなどのナレッジ管理ツールでは `[[ファイル名]]` という記法でファイル間リンクを行う。
これらのツールで作成したリポジトリを閲覧する際、Wikiリンクが機能しないとナビゲーションが不便である。

2026-07-10に設計論点を確定して着手。renderer-plugins specで導入したプラグイン機構の最初の追加事例として、`plugins/wiki-links/` として実装する（境界基準「GFM相当＝コア、それ以外の表現拡張＝プラグイン」に基づく）。

## スコープ

- **対象**: `[[ページ名]]`・`[[ページ名|表示テキスト]]`・`[[ページ名#見出し]]`（および同一ファイル内 `[[#見出し]]`）のパース、ワークスペース内でのリンク先解決、クリックによる遷移、未解決リンクの視覚的フィードバック
- **対象外**:
  - 埋め込み `![[ページ名]]`（別ファイル内容の展開表示）→ 再帰・パフォーマンス考慮が必要なためv2以降
  - ブロック参照 `[[ページ名^ブロックID]]`
  - リンク先ファイルの新規作成（ビューワーのため編集機能は対象外）
  - バックリンク一覧・オートコンプリート

## ユーザーストーリー

### US-001: Wikiリンクのパースと遷移

```
As a ユーザー
I want to Markdown内の [[ページ名]] をクリックして対象ファイルを開く
So that ナレッジベース内のページ遷移をスムーズに行える
```

**受け入れ条件**

- [x] `[[ページ名]]` が通常のリンクと同じ見た目でレンダリングされる
- [x] `[[ページ名|表示テキスト]]` は表示テキストで描画される
- [x] `[[ページ名#見出し]]` はリンク先ファイルを開いた後、該当見出しへスクロールする
- [x] `[[#見出し]]` は同一ファイル内の該当見出しへスクロールする
- [x] クリックすると、ワークスペース（エクスプローラーのルート）内から該当ファイル名のMarkdownを解決して開く
- [x] インラインコード・コードブロック内の `[[...]]` はリンク化されない

### US-002: リンク先の解決ルール

```
As a ユーザー
I want to Obsidianと近い規則でリンク先が決まる
So that 既存のナレッジベースを違和感なく閲覧できる
```

**受け入れ条件**

- [x] ページ名は拡張子なしのファイル名（大文字小文字は区別しない）でマッチする。`.md`/`.markdown` 拡張子付きの指定も可
- [x] `[[フォルダ名/ページ名]]` のようなパス付き指定はパス末尾一致で解決される
- [x] 同名ファイルが複数ある場合、現在ファイルに近いもの（同一フォルダ→近いフォルダの順）を優先する
- [x] 探索は `respectGitignore` 設定に従う（ツリー・全文検索と同一の可視範囲）
- [x] フォルダを開いていない場合は現在ファイルの親フォルダ配下を探索範囲とする

### US-003: 未解決リンクのフィードバック

```
As a ユーザー
I want to リンク先が存在しない [[ページ名]] がひと目で分かる
So that 壊れたリンクをクリックして無駄なエラーに遭遇しない
```

**受け入れ条件**

- [x] リンク先が見つからないWikiリンクはグレーアウト表示になる
- [x] グレーアウトリンクはクリックしても何も起きない（エラーダイアログを出さない）
- [x] ホバーで「リンク先が見つかりません」と表示される

## 未決定事項（設計フェーズで決定）

- [x] 対応構文の範囲 → 基本 + エイリアス + 見出しアンカー。埋め込み `![[...]]` はv2以降
- [x] 同名ファイル複数時の優先順位 → 現在ファイルに近い順（Obsidian互換）
- [x] 探索時の.gitignore考慮 → `respectGitignore` 設定に従う
- [x] ルート未設定時の挙動 → 現在ファイルの親フォルダ配下を探索
