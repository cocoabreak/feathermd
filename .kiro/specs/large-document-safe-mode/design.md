# 技術設計: 大容量文書のセーフモード

## ステータス

完了

## 方針

セーフモードは「機能を減らしたMarkdownレンダラー」ではなく、Markdownソースをプレーンテキストとして表示する専用経路とする。通常の`markdown-it`処理を部分的に残すと、5MiB文書で約25万DOM要素を生成する問題が残るためである。

既存の5MiB警告閾値と10MiB読込上限は変更しない。最初の確認では「セーフモードで開く／キャンセル」の二択とし、通常表示はセーフモードで内容を確認した後の明示操作として提供する。これにより、ネイティブ二択ダイアログを維持しながら、通常レンダリングまでを二段階の操作にする。

## 表示モード

```ts
type RenderMode = "full" | "safe";
```

表示モードはタブの実行時状態として保持する。既存タブの既定値は`full`とし、永続化形式には追加しない。セッション復元ではファイルを再読込するため、大容量文書は再びセーフモード確認を通る。

`MarkdownViewer`は`renderMode`に応じて経路を分ける。

- `full`: 既存のMarkdownエンジン、サニタイズ、プラグイン、DOM後処理を使用する
- `safe`: Svelteのテキスト補間で`<pre>`相当の専用コンポーネントへソースを渡し、`{@html}`を使用しない

セーフモードでは本文をプレーンテキストとして扱う。行単位の`span`等は生成せず、DOM数を入力行数へ比例させない。目次ジャンプ用にRustが抽出した見出し位置だけ空のアンカーを挿入するが、最大2,000件に制限する。CSSは`white-space: pre-wrap`と`overflow-wrap: anywhere`を使用する。

## Rustによるセーフモード目次

大容量文書に通常の`markdown-it`レンダリングを適用せず目次を提供するため、Rust側へ`pulldown-cmark 0.13.4`を追加する。Sätteriも検討したが、本アプリはRustバックエンドから直接crateを利用でき、MDXやJavaScriptプラグイン境界を必要としないため、基盤の`pulldown-cmark`を直接採用する。

`read_file`は5MiB以上の本文だけをCommonMark/GFMとして走査し、次を`safeOutline`として返す。

- 見出しレベル
- Text / Codeイベントを連結した表示文字列
- `safe-heading-<index>`形式の一意ID
- JavaScript文字列で使用できるUTF-16オフセット

UTF-8バイトオフセットは見出し順に本文を一度だけ走査してUTF-16へ変換し、日本語やサロゲートペアを含む本文でもSvelte側の`slice`位置と一致させる。見出しは先頭2,000件までとし、2,001件目を検出した時点で走査を終了して`safeOutlineTruncated`を返す。

`SafeModeView`は本文を見出しオフセットで分割し、各位置へ空のアンカーを挿入する。本文そのものはSvelteのテキスト補間を維持し、`{@html}`は使用しない。通常表示のTOCと同じUIを使い、クリック時はアンカーへ`scrollIntoView`する。

## オープンフロー

既存の`readMarkdownFile`が返す`byteSize`と`requiresConfirmation`を利用する。

1. 小容量文書は従来どおり`full`で開く
2. 大容量文書で通常表示が実行中承認済みなら`full`で開く
3. 未承認の大容量文書は、ネイティブ確認でセーフモードを案内する
4. 承認時は`safe`、拒否時はキャンセルとして返す
5. タブ作成側は本文と表示モードを同時に反映する

読込アクションの返却値は、本文だけではなく表示判断を含む構造へ変更する。

```ts
type LoadedMarkdown = {
  raw: string;
  byteSize: number;
  renderMode: RenderMode;
  safeOutline: SafeOutlineHeading[];
  safeOutlineTruncated: boolean;
};
```

通常表示の実行中承認Setは既存の大容量承認状態を引き継ぐ。セーフモードで開いただけではSetへ追加しない。

`readMarkdownFile`は新規オープンだけでなく、セッション復元とwatcher再読込からも使われる。呼び出し側が現在のモードを渡せるよう、次の責務を持たせる。

```ts
readMarkdownFile(path, {
  currentMode?: RenderMode;
  reason?: "open" | "watch";
}): Promise<LoadedMarkdown | undefined>
```

- `currentMode: "safe"`: サイズが小さくなっても`safe`を維持し、確認を表示しない
- `currentMode: "full"`: 小容量または通常表示承認済みなら`full`を維持する
- `currentMode: "full"`かつ`reason: "watch"`の未承認パスが5MiB以上になった場合: 確認を表示せず`safe`へ移行する
- `currentMode`なし: 新規オープン・復元として通常のサイズ判定を行う

`LoadedMarkdown`の`renderMode`は、本文を`contentStore`へ保存する処理と同じ成功経路でタブへ反映する。本文だけ先に更新してモードが古いままになる部分状態を作らない。

## 通常表示への切り替え

セーフモード案内に「通常表示」アクションを設ける。選択時はネイティブ確認を表示し、応答停止の可能性を改めて伝える。

承認時はパスを通常表示承認Setへ追加して同じタブの`renderMode`を`full`へ変更する。拒否またはダイアログエラー時は`safe`を維持する。別タブや別パスへ承認を波及させない。

通常表示からセーフモードへ戻す操作は本specの対象外とする。通常表示の処理中はWebViewが応答不能になり得て、切り替え操作を保証できないためである。

## watcher・履歴・タブ

- watcher再読込は対象タブの現在の`renderMode`を優先する
- `safe`タブの更新では再確認せず、更新後も`safe`で表示する
- 実行中承認済みの`full`タブは更新後も`full`で表示する
- 未承認の`full`タブが更新によって大容量になった場合、確認なしで本文とモードを`safe`へ更新する。継続書き込みによる確認ダイアログの連発を防ぎ、通常表示は本文バナーからの明示操作に限定する
- Back/Forwardで既存タブ内のファイルが変わる場合は、新しいパスに対して通常のサイズ判定を行う
- 最近使ったファイルへの追加は、セーフモードでタブ作成に成功した後に行う

## 派生状態と利用可能な操作

`MarkdownViewer`が`safe`へ入る際は、通常表示のレンダリング世代を無効化し、DOM後処理のcleanupを実行する。TOC、active heading、frontmatter、読書統計、検索結果と検索ハイライトを空にし、前のタブや切替前の通常表示から派生状態を持ち越さない。

セーフモード案内は本文領域の先頭へ配置する。通常表示ボタンへキーボードで到達でき、案内文は支援技術から読める構造にする。StatusBarだけには置かない。

既存のページ内検索は一致箇所ごとに`mark`要素を生成するため、セーフモードでは無効化する。検索UIを開こうとした場合は、セーフモード中は利用できないことを通知する。大容量文書向け検索は後続の「検索結果UI」specで、結果件数上限とDOM非依存の走査を含めて設計する。

印刷は画面上のプレーンテキストを対象にしてよい。HTML、SVG、PNG等、通常レンダリング済みDOMを前提とするexportとコンテキストメニュー項目は無効化し、切替前の`renderedHtml`を出力しない。

## セキュリティ・プライバシー

セーフモードは本文をテキストとしてのみ表示し、次を行わない。

- `{@html}`によるDOM挿入
- `img`、`a`、`iframe`等の要素生成
- 外部HTTPS画像・リンクへの通信
- ローカル画像解決やRust画像読込コマンド
- Mermaid、KaTeX、Shikiの動的import
- レンダラープラグインとDOM post-process

ファイル読込自体は従来の`AllowedRoots`、最終ハンドルパス検証、Markdown拡張子制限、10MiB上限を通す。

## エラー処理

- 最初の確認拒否はユーザーキャンセルとして扱い、エラー通知しない
- 最初の確認ダイアログが失敗した場合はキャンセルとして扱い、タブやキャッシュを変更しない
- 同一パスへ複数の新規オープン要求が並行した場合、進行中の確認Promiseを共有してダイアログを重複表示しない
- セーフモード表示中の通常表示確認拒否は表示モードを変更しない
- ファイル再読込失敗時は既存のエラー表示を使用し、直前の安全な本文を不正な部分状態へ置き換えない
- 表示モードが欠落・不正な場合は`full`へ暗黙フォールバックせず、既存タブでは`full`、大容量読込結果では`safe`を明示的に設定する

## テスト

- action: 小容量=`full`、大容量承認=`safe`、拒否=キャンセル、通常表示承認Setとの分離
- store/tab: モードのタブ別保持、切替、クローズ、watcher更新
- component: セーフモードでソースがテキスト表示され、HTML・画像・リンク要素が生成されない
- component: 通常表示確認の承認・拒否とモード切替
- regression: 小容量文書の通常表示、セッション復元、リンク、検索、watcher
- performance: 5MiB・10MiB文書をrelease相当WebViewで開き、表示時間、DOM数、操作継続可否を記録する
- security: 外部画像、HTML、Mermaid、ローカル画像参照を含む大容量文書で通信・コマンド呼出しが発生しない

## 性能計測結果

2026-07-12にWindows 11のTauri devビルド（WebView2）で、CDPからタブと本文ストアへセーフモード状態を設定し、本文のテキストノード反映までを計測した。

| 入力             |          テキストDOM反映 | アプリ全体のDOM要素数 | 危険要素             |
| ---------------- | -----------------------: | --------------------: | -------------------- |
| 5,600,084 bytes  | 画面表示と操作継続を確認 |                   117 | `img` / `a` 生成なし |
| 10,485,626 bytes |                     27ms |                   124 | `script`生成なし     |

10MiB相当の本文も単一`pre`内のテキストとして全量が反映され、DOM要素数は入力行数に比例しなかった。`Ctrl+F`では検索入力を生成せず、セーフモード中は利用できない旨の通知を表示した。

この計測はセーフ表示経路の性能とDOM生成を対象にしており、ネイティブ確認ダイアログ、Rustの実ファイル読込、watcherによる自動切替、通常表示への切替は別途実アプリで確認する。

### pulldown-cmark PoC

同一の見出し・段落・強調・リンク・リスト・引用・コードブロックを繰り返すASCII文書を用い、release設定で各5回実行した中央値を比較した。

| 入力  | pulldown-cmark HTML | markdown-it基本HTML | Rust目次抽出 |
| ----- | ------------------: | ------------------: | -----------: |
| 1MiB  |                28ms |               206ms |         24ms |
| 5MiB  |               138ms |               984ms |        125ms |
| 10MiB |               286ms |             2,057ms |        247ms |

HTML生成は約7倍高速だったが、10MiB入力から約19MiBのHTMLが生成され、大量DOM問題は解消しない。そのため全文HTMLへの置換は行わず、目次抽出だけに採用した。

Windows 11のTauri devビルドで5,500,022 bytes・2,000見出しの実ファイルをRust `read_file`経由で確認した。ファイル読込と目次抽出は529ms、セーフ本文とアンカーのDOM反映は144ms、アプリ全体のDOM要素数は4,121だった。目次クリックから該当ソース位置へのスクロールも確認した。releaseビルドの目次抽出単体は5MiBで125msであり、debug計測より短い。
