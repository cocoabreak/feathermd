# 要求定義: 安全性の向上 (security-hardening)

## 背景・動機

現状 `sanitizeHtml`（DOMPurify）によりレンダリング後HTMLのXSSは対策済みだが、それ以外のローカルファイルアクセス経路が検証不足のまま残っている。

特に `app/src-tauri/tauri.conf.json` の asset protocol scope が `["**"]` になっており、**PC上の任意のパスをasset protocol経由で読み出せる状態**になっている。悪意あるMarkdownファイル（メール添付・ダウンロード等で入手したもの）を開いた場合、`![x](../../../../Users/<user>/.ssh/id_rsa)` のような相対パスの画像参照や、ローカルファイルリンクの`../`を辿ることで、開いたフォルダ/ファイルの外側にある任意のファイルにアクセスできてしまう。これは緊急度の高い既存の脆弱性であり、本specの中で最優先に修正する。

## スコープ

- **対象**:
  - ローカルファイルリンク（Markdown内の相対パスリンク）の参照範囲をルート配下に制限
  - ローカル画像参照の参照範囲をルート配下に制限
  - 画像読み込み方式の変更（asset protocolワイルドカード撤去 → パス検証を伴うRust側コマンド経由でのbase64データURL化）
  - 上記に伴う `tauri.conf.json` / `capabilities/default.json` の権限スコープ見直し
- **対象外**: 既存のHTMLサニタイズ（DOMPurify）自体の見直し、CSP設定の変更、Kroki等の外部ネットワークリクエストに関する対策、依存関係の脆弱性監査（別途 `RUSTSEC-2026-0194`/`0195` のIssueで追跡中）

---

## ルート（許可範囲）の定義

| 開き方 | ルート |
|---|---|
| フォルダを開く（エクスプローラーでルート指定） | 開いたルートフォルダ配下 |
| ファイル単体を直接開く | そのファイルの親ディレクトリ配下 |

- ルートの外側への参照（`../` 等でルートを越えるパス）は解決を拒否する
- 複数タブでそれぞれ異なるファイル/フォルダを開いている場合、ルートはタブ（＝開いた文脈）ごとに独立して判定する

## ユーザーストーリー

### US-001: ローカルファイルリンクの参照範囲制限

```
As a ユーザー
I want to 開いたルートの外側にあるファイルへのリンクを無効化する
So that 悪意あるMarkdownファイルによって意図しないローカルファイルの存在を探索されない
```

**受け入れ条件**

- [x] ルート配下の相対パスリンクは従来どおり開ける
- [x] `../` を含み解決結果がルートの外側になるリンクはクリックしても開かず、エラー表示する
- [x] 絶対パスリンク（`C:/...` 等）でルート外を指す場合も同様に拒否する

---

### US-002: ローカル画像参照の範囲制限とbase64化

```
As a ユーザー
I want to 開いたルートの外側にある画像ファイルを読み込ませない
So that asset protocolの広範な許可を悪用した任意ファイル読み出しを防ぐ
```

**受け入れ条件**

- [x] `tauri.conf.json` の asset protocol scope の `["**"]` ワイルドカードを撤去する
- [x] 画像はRust側の専用コマンドでルート内チェックを行った上で読み込み、base64データURLとしてフロントエンドへ返す
- [x] ルート配下の画像は従来どおり表示される
- [x] ルート外を指す画像パスは読み込まれず、代替表示（壊れた画像アイコン等、ブラウザ標準の挙動でよい）になる
- [x] `http://` / `https://` / `data:` で始まる画像srcは対象外（変更なし、現状維持）

---

### US-003: 権限スコープの最小化

```
As a 開発者
I want to Tauriのcapabilities/asset protocol設定を実際に必要な範囲まで絞る
So that 想定外のファイルアクセス経路が残らない
```

**受け入れ条件**

- [x] asset protocol scopeのワイルドカード除去後も、通常のMarkdown閲覧・画像表示に機能的な影響がない
- [x] `capabilities/default.json` の既存権限（fs:allow-read-file 等）についても、今回の変更で不要になるものがあれば削除する

---

## 非機能要求

| 項目 | 要求 |
|------|------|
| 互換性 | 既存のMarkdownファイル（ルート配下の相対パス画像・リンク）の表示に回帰がないこと |
| パフォーマンス | base64化によるオーバーヘッドが、通常サイズの画像（数百KB程度）で体感できるほど悪化しないこと |

## 未決定事項（設計フェーズで決定）

- [x] ルート外参照を拒否した際のエラー表示方法（alert / インライン表示 / コンソールのみ）
- [x] base64化コマンドの具体的なインターフェース（引数・戻り値の型）
- [x] シンボリックリンクでルート外に実体があるファイルの扱い
