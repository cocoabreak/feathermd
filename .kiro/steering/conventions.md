# 開発規約

## 改行コード

- **LF** に統一する。
- Git設定: `core.autocrlf = input`、`core.eol = lf`（リポジトリ初期化時に適用済み）。
- 新規ファイル作成時は必ずLFで保存する。`.gitattributes` で強制している。

## コミュニケーション

- 日本語で応答する（コード・変数名は英語）。
- 簡潔に回答し、自明な説明は省略する。
- 複雑なタスクでは実装前に計画を提示し、承認後に着手する。

## Git規約

- **ブランチ戦略**: GitHub Flowを採用。`main` ブランチへ直接コミットせず、必ず機能ブランチを切る。
- **ブランチ命名規則**:
  - `feat/xxx`: 新機能開発
  - `fix/xxx`: バグ修正
  - `docs/xxx`: ドキュメント整備
  - `refactor/xxx`: リファクタリング
  - `chore/xxx`: ビルド設定や依存ライブラリの更新
- **コミットメッセージ**: Conventional Commits形式、本文は日本語。
  - 例: `feat: Markdownファイルのプレビュー機能を実装`
- **マージ方針**: Squash Mergeを推奨。節目でGit Tagを付与する。
- **マージ前のspecステータス確認**: 機能ブランチを`main`へマージする前に、対象の`.kiro/specs/<feature>/`について以下を確認し、完了しているものは確定させてからマージする。
  - `tasks.md`: 実装済みの項目は`[x]`、v1スコープ外と判断した項目は`[-]`にする（`[ ]`が実装完了後も残らないようにする）。
  - `requirements.md`の「未決定事項（設計フェーズで決定）」: `design.md`で確定済みなら`[x]`にする。v2以降へ持ち越す項目はチェックせず、その旨を注記する。
  - `design.md`の「ステータス」: 未決定事項がすべて確定していれば「完了」にする。
  - `.kiro/backlog.md`: 保留・候補として載せていたアイデアがspec化・実装済みになった場合、該当エントリを「実装済み（参考）」セクションへ移す（またはそのspecへの参照を追記する）など、記載内容がspecの実態と矛盾しないようにする。
- 確認なしに自動コミット・自動pushしない。
- 依存関係のない並列作業が必要な場合のみ、Git WorkTreeを検討する。

## 基本原則

- DRY (Don't Repeat Yourself) の徹底。
- KISS (Keep It Simple, Stupid)。
- 単一責任原則 (SRP)。
- マジックナンバーの回避（定数化）。

## Serena MCPの利用

Serena MCPが利用可能な環境では、Language Serverによる意味解析をコード調査・変更に活用する。

- コーディングタスクの開始時にSerenaの初期指示を読み、プロジェクトを有効化する。オンボーディング未実施の場合は先に完了する。
- シンボルの定義・参照・実装・呼び出し元を調べる場合は、全文検索より先にSerenaのシンボル検索・参照検索を検討する。
- シンボルの変更前に参照元を確認し、公開APIや複数レイヤーに影響する変更では影響範囲を報告する。
- 関数・型などシンボル全体の置換、リネーム、安全な削除にはSerenaのシンボル編集機能を優先する。数行だけの変更、設定・Markdown、複数シンボルにまたがる機械的変更は通常のパッチ編集を使う。
- 変更後はSerenaの診断を補助的に確認する。ただし、プロジェクト所定のformat / lint / check / testを省略しない。
- Rustではシンボル解析を積極的に利用する。Svelte/TypeScriptはLanguage Serverが十分なシンボルを返さない場合があるため、その場合は`rg`による検索、型チェック、既存テストを併用する。
- Serenaのmemoryには、長期的に有効で再発見コストが高いプロジェクト知識だけを保存する。一時的な調査結果、個別タスクの進捗、実在するローカル絶対パスは保存しない。
- `.serena/project.yml`と`.serena/memories/`は共有可能なプロジェクト設定・知識として管理し、`.serena/cache/`と`.serena/project.local.yml`はローカル専用としてコミットしない。

## 禁止事項

- README・ドキュメントを確認なしに生成・変更しない。
- テストコードを確認なしに削除・コメントアウトしない。
- 既存の動作するコードを理由なくリファクタリングしない。
- **個人PCのローカル環境情報をリポジトリに記録しない。** 実在する絶対パス（例: `E:\Development\...`、`C:\Users\<実ユーザー名>\...`）やOSユーザー名を、spec・ADR・ドキュメント・コード・テスト・コミットメッセージに書かない。例示が必要な場合は架空のパス（`D:/projects/notes`、`C:\Users\alice` 等）やプレースホルダー（`<dev-root>` 等）を使う。リポジトリ名など環境に依存しない識別子での言及は可。

## コーディング規約

### フォーマット・静的解析（必須）

コードをコミットする前に、以下のツールを必ず実行すること。

**フロントエンド（`app/` ディレクトリ）:**

```bash
npm run format   # Prettier による自動フォーマット
npm run lint     # ESLint による静的解析
npm run check    # svelte-check による型チェック
```

**Rustバックエンド（`app/src-tauri/` ディレクトリ）:**

```bash
cargo fmt        # rustfmt による自動フォーマット
cargo clippy     # Clippy による静的解析
```

- `format` / `cargo fmt` はエラーが出なくなるまで適用する。
- `lint` / `cargo clippy` の警告・エラーは修正してからコミットする。
- `check` のエラーはコミット前に必ず解消する。警告も原則解消すること。

## レビューワークフロー（必須）

レビューは実装者と異なる視点を得るための工程であり、フォーマッター・静的解析・テストの代替にしない。

### 実施順序

1. 実装と必要なテストを完了する。
2. format / lint / check / test / cargo fmt / cargo clippy と必要な実アプリ確認を完了する。
3. 下記の起動条件に従いレビューを実施する。利用可能なら別サブエージェントへ該当スキルを割り当て、レビュー担当は読み取り専用とする。
4. 指摘を修正した場合、影響する検査・動作確認・レビューを再実施する。
5. 必須レビューの未解決P0/P1/P2がないことを確認してからコミット・マージする。P3を見送る場合は理由を報告する。

### 設計・差分レビュー

- 使用スキル: `.agents/skills/review-design-diff/`
- 機能追加、バグ修正、リファクタリング、依存関係変更、永続化形式変更、複数レイヤーにまたがる変更では必須。
- specやADRの設計判断を変更するドキュメント更新でも必須。
- 誤字修正、コメントのみ、機械的フォーマット、生成物のみの変更は省略可。
- 仕様整合、既存設計との整合、回帰、ライフサイクル、エラー処理、テスト不足を確認する。Prettier等が検出する事項を重複確認しない。

### セキュリティレビュー

- 使用スキル: `.agents/skills/review-security/`
- 次のいずれかに触れる変更では必須: ファイルシステム、パス・信頼済みルート、HTML/SVG/CSS生成・サニタイズ、Tauriコマンド・capability・CSP、外部URL・外部プロセス、プラグイン、永続化された入力、CLI引数、ドラッグ＆ドロップ、watcher、エクスポート、新規依存関係。
- 上記に該当しない表示文言、純粋なレイアウト、内部計算のみの変更では省略可。
- 信頼境界、入力検証、権限、注入、パストラバーサル、リソース枯渇、TOCTOU、fail-closedを重点確認する。

### レビューの独立性

- レビュー担当には現在の差分、対象spec、検査結果を渡す。実装者の結論や「問題ないはず」という誘導は渡さない。
- レビュー担当は原則コードを変更せず、重大度、根拠、影響、最小の修正方向を報告する。
- 小さな試作中はレビューを後回しにしてよいが、コミット・マージ前には必ず起動条件を評価する。
